# 差旅搭子（Travel Buddy）需求规划文档

> 阶段一交付物 · 需求规划  
> 版本：v1.0 · 适用范围：桌面端（Python 3.11 + PySide6 + SQLite）  
> 文档目标：在立项开发前，明确用户、痛点、场景、目标、竞品、需求池与待确认决策，作为后续设计/开发/测试用例的源头依据。

---

## 1. 目标用户画像

| 用户类型 | 角色描述 | 出差频率 | 核心痛点 |
|---|---|---|---|
| **画像 A：高频差旅销售/商务** | 一线销售、BD、商务拓展，月均出差 3–6 次，跨城高铁/飞机为主，餐饮打车零散票多 | 高（几乎每周） | 发票量大、来源杂（邮件 PDF、微信文件、纸质拍照），月底集中报销时归集耗时，易漏票、易重报 |
| **画像 B：项目驻场/实施顾问** | 软件实施、审计、咨询，长期驻外 1–3 个月，酒店+交通为主，需按项目/客户归集 | 中高（阶段性集中） | 发票需对应到具体项目/行程做成本归集；手工 Excel 难按"行程+分类"多维统计，月底与财务对账费劲 |
| **画像 C：行政/财务对接人（轻量使用者）** | 部门助理或财务对接人，替团队代收/整理发票，导出报销明细表提交财务 | 中（周期性） | 缺乏统一格式，接收的是同伴散乱文件；需要一键导出规范 Excel/PDF 报销单，减少反复格式校验 |

> 设计取舍：阶段一以**个人单机使用**优先（A、B 为主），C 类作为导出能力的受益方，暂不做团队协同。

## 2. 用户痛点清单（按严重度标注）

严重度说明：**P0 致命/必须解决**（不解决产品无价值）、**P1 高**（显著影响留存）、**P2 中**（影响体验效率）、**P3 低**（锦上添花）。

| 编号 | 痛点 | 严重度 | 说明 |
|---|---|---|---|
| PN-1 | 发票散落在邮件/微信/本地，报销前难统一归集 | P0 | 多源分散是立项核心原因，无归集则产品价值不成立 |
| PN-2 | 手工从 PDF/图片抄录字段（金额、税号等）效率低且易错 | P0 | 解析自动化是核心差异点，纯手工录入无竞争力 |
| PN-3 | 月底集中找票、担心漏报/重报 | P1 | 需"状态机 + 搜索筛选"支撑，避免遗漏与重复提交 |
| PN-4 | 无法按"行程/分类"做成本归集与对账 | P1 | 画像 B 强诉求，关联 Trip 实体是关键能力 |
| PN-5 | 导出的报销表格式不统一，财务反复打回 | P1 | 需规范 Excel 模板 + PDF 报销单，降低返工 |
| PN-6 | 隐私担忧：发票含税号等敏感信息，不愿上传云端 | P1 | 本地 SQLite + 附件落盘是卖点，需明确"不上云"承诺 |
| PN-7 | 图片类发票（纸质拍照）无法解析，只能手工 | P2 | 依赖 OCR，能力可选，但影响图片场景覆盖率 |
| PN-8 | 缺乏可视化概览，不清楚本月花了多少、待报销多少 | P2 | 仪表盘满足感知与规划需求 |
| PN-9 | 邮件导入配置复杂（IMAP/授权码），普通用户不会配 | P2 | 需引导式配置与默认值，降低门槛 |
| PN-10 | 与现有财务系统/OA 报销流程割裂，仍需二次录入 | P3 | 阶段一不打通，但需在路线图说明 |

## 3. 核心使用场景（端到端）

### 场景 S1：本地批量导入 + 自动解析（最高频）
1. 用户打开"导入"→ 选"本地文件"，多选 20 张高铁票 PDF + 手机拍的 5 张餐饮小票图片。
2. 系统对 PDF 用 pdfplumber 抽取文本并正则解析字段；图片尝试 OCR（若启用）。
3. 解析结果按规则自动归类（火车票→train、餐饮→meal），摘要列表展示，识别失败项标黄。
4. 用户逐条核对/修正金额与税号，确认后批量入库，状态默认"待报销"。
5. 在"集中管理"中按分类查看，总数与合计金额实时更新。

### 场景 S2：邮件自动归集（IMAP 拉取）
1. 用户在"设置"中配置 QQ 邮箱 IMAP 与授权码（引导式，含常见主机下拉）。
2. 点击"从邮件导入"，系统拉取近 30 天带 PDF/图片附件的邮件。
3. 自动去重（按发票代码+号码），解析后归入"待报销"，来源标记为"邮件导入"。
4. 用户筛选"来源=邮件"复查，将某几张挂到对应"上海出差"行程。

### 场景 S3：按行程归集与对账（画像 B）
1. 用户新建行程"上海客户A项目"，填目的地/起止日/预算。
2. 把该期间导入的机票、酒店、打车发票批量关联到该行程。
3. 在"仪表盘→行程汇总"看到该行程已归集金额 vs 预算，分类占比一目了然。
4. 出差结束前核对有无漏票，对超预算项提前预警。

### 场景 S4：一键导出报销单（画像 C / 月底）
1. 用户在"集中管理"勾选本次要报的发票（或按行程/状态筛选后全选"待报销"）。
2. 点"导出"→ 生成 Excel 报销明细表（含发票代码、号码、金额、税额、销售方等字段）。
3. 同时生成 PDF 报销单（封面+明细），附件（原始 PDF/图片）可一并打包。
4. 提交财务，系统将选中发票状态置为"已提交"，避免重复导出。

### 场景 S5（可选增强）：链接导入
复制同事发的电子发票 URL → 粘贴到"链接下载"→ 系统 requests 下载到本地附件目录并走解析流程，来源标记"链接下载"。

## 4. 业务目标

### 定性目标
- 成为频繁出差职场人"发票归集第一站"，用"本地优先 + 自动解析"建立隐私与效率信任。
- 把月底报销准备时间从"数小时"压到"分钟级"，减少漏报/重报。
- 通过规范导出降低与财务的沟通返工。

### 定量目标（阶段一验收参考）
| 指标 | 目标值 | 说明 |
|---|---|---|
| 单批导入解析字段准确率（PDF） | ≥ 95% | 抽样 200 张常见发票，关键字段自动填充正确率 |
| PDF 解析耗时 | 单张 < 1.5s | 含文本抽取+正则，批量可并行 |
| 月底报销准备耗时 | 较手工 ↓ 70% | 用户调研对比 |
| 漏报/重报发生率 | 趋近于 0 | 靠状态机 + 去重 |
| 支持的发票来源 | ≥ 3 种（本地/邮件/手工，链接为 Should） | 覆盖主要入口 |
| 导出成功率 | 100% 可打开、字段完整 | Excel + PDF |

> 注：定量目标需在阶段二（设计/开发）转化为可测的验收标准。

## 5. 竞品调研与现有方案对比

> 来源说明：以下基于**公开产品功能认知与已知信息**（微信发票管家、钉钉/飞书报销、Excel 手工管理、票据类 App 的公开功能），未做实时联网实测；上线前建议补做一次竞品实测校准。

| 方案 | 功能覆盖 | 优点 | 缺点 | 与差旅搭子的差异 |
|---|---|---|---|---|
| **微信发票管家** | 微信内电子发票归集、转发到企业微信/卡包 | 入口天然在微信、用户量大 | 仅限微信生态、依赖上传、企业报销需对接、非桌面、不支持本地图片批量、无差旅行程归集 | 本产品为**桌面端、本地存储、支持 PDF/图片/邮件多源、按行程归集**，隐私与差旅场景更强 |
| **钉钉/飞书报销** | OA 内填单、审批流、发票上传、对接财务 | 流程闭环、企业统一、可审批 | 强依赖企业开通、需联网上传、表单化录入繁琐、个人零散票管理弱、非单机隐私 | 本产品定位"报销前归集"前置环节，可导出对接，不替代 OA，强调本地与个人隐私 |
| **Excel 手工管理** | 自由表格记录 | 零门槛、灵活 | 全手工抄录易错、无解析、无去重、无统计图表、难按行程 | 本产品以自动解析+仪表盘+导出模板替代手抄，效率质变 |
| **票据类 App（如发票管家、扫描全能王等）** | 手机拍照识别、发票查验、部分支持导出 | 移动端 OCR 成熟、便携 | 多为手机端、跨端同步需账号、导出格式有限、桌面深度管理弱、部分收费 | 本产品主打**桌面深度管理 + 多源 + 行程/分类三维组织 + 本地隐私**，互补而非直接替代 |
| **本产品 差旅搭子** | 多源采集+智能解析+三维管理+仪表盘+规范导出+本地化 | 本地隐私、差旅场景聚焦、自动解析、行程归集、导出规范 | 阶段一无 OCR/无云端同步/无 OA 打通（见待确认） | — |

**结论**：市场缺一个"桌面端、本地优先、面向差旅场景的发票归集+导出"工具；本产品以隐私与差旅归集切入，与现有方案形成互补而非硬碰硬。

## 8. 优先级建议（MoSCoW 法）

| 分级 | 含义 | 包含需求 |
|---|---|---|
| **Must（必须有，阶段一上线）** | 无则产品不成立 | R-01 本地导入、R-04 手工录入、R-05 PDF 解析、R-06 自动分类、R-07 结果可编辑、R-09 三维管理、R-10 状态机、R-11 行程管理、R-14 Excel 导出、R-16 本地持久化 |
| **Should（应该有，阶段一尽量做）** | 显著提升价值 | R-02 邮件导入、R-03 链接导入、R-12 去重、R-13 仪表盘、R-15 PDF 报销单、R-17 设置、R-18 邮件引导配置 |
| **Could（可以有，资源富余再做）** | 增强体验 | R-08 图片 OCR 解析（依赖外部引擎，复杂度高） |
| **Won't（阶段一不做）** | 明确排除，留待后续 | 多用户/团队协同、OA/财务系统对接、云端同步、移动端 |

> 价值-复杂度矩阵速读：高价值低复杂度的 Must 项（解析、导出、本地化、三维管理）优先投入；高复杂低确定性的 Could/Won't 项（OCR、协同、对接）后置，避免阶段一范围蔓延。

## 9. 待确认问题清单（关键决策缺口）

| 编号 | 待确认问题 | 影响 | 建议决策方 |
|---|---|---|---|
| Q-1 | **OCR 是否必须做？** 图片类发票（纸质拍照）在阶段一是否纳入解析，还是仅支持手工录入？ | 决定 R-08 范围与依赖（如 paddleocr/tesseract） | 产品+技术 |
| Q-2 | **是否需要多用户/团队协同？** 还是严格单机个人版？ | 决定数据模型与架构（是否上账号/同步） | 产品负责人 |
| Q-3 | **报销单/Excel 格式规范？** 是否需对齐公司财务模板，还是通用模板即可？ | 决定 R-14/R-15 字段与版式 | 财务对接人 |
| Q-4 | **邮件导入的授权与隐私边界？** IMAP 授权码存储方式（明文/系统钥匙串）？ | 安全合规 | 技术+安全 |
| Q-5 | **OFD 格式支持深度？** 仅存储还是需解析（OFD 需专门库 ofdparser）？ | 影响解析范围与依赖 | 技术 |
| Q-6 | **发票查验（真伪/查重对接国税）是否阶段一做？** | 决定是否与外部 API 集成 | 产品 |
| Q-7 | **数据备份/迁移策略？** 单文件 SQLite 如何备份、换机迁移？ | 用户体验与数据安全 | 产品+技术 |
| Q-8 | **自动归类规则的准确率预期与可配置性？** 用户能否自定义分类与归类规则？ | 决定 R-06 灵活度 | 产品 |
| Q-9 | **图片附件的存储与体积控制？** 是否压缩、原图留存策略？ | 磁盘占用 | 技术 |
| Q-10 | **是否需要发票与"公司/税号"绑定（多购买方）？** 多人共用一台机器场景？ | 模型与筛选 | 产品 |

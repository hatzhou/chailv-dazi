## 5. 错误根因分析（按字段拆解）

### 5.1 金额 `amount` — 命中率 20%（最严重，影响 8/10 张）

当前版正则只认三种模式，但真实发票四种版式全不匹配：

| 发票类型 | 真实文本特征 | 当前版为何失败 |
|---|---|---|
| **铁路电子客票**（01~04, 07） | `￥276.00\n票价:` — 金额在"票价"**之前** | 当前版只搜 `价税合计/合计/金额` 标签，不认识"票价" |
| **增值税专用发票**（06） | `价税合计（大写）伍佰壹拾陆圆整 （小写）¥516.00` — 标签与数字被**大写金额**隔断 | 正则要求标签紧跟数字，被 `（大写）…` 阻断 |
| **增值税普通发票**（05） | `合 计 ¥198.80` — 标签带空格 | 当前版搜的是紧凑 `合计`，空格版漏掉 |
| **客运票**（08） | 价税合计在**表格行内** `¥39.80`，与标签行分离 | 正则扫描的是标签行附近，距离太远 |

唯一命中的是滴滴行程单（`合计45.00元` 和 `合计33.70元`）——恰好符合 `合计 + 数字` 的简单模式。

### 5.2 购买方名称 `bname` — 命中率 0%（全部被污染）

- 铁路票文本为 `购买方名称:信雅达科技股份有限公司 统一社会信用代码:91330000253917765N`，当前版 `_parse_name` 的截断逻辑用 `re.split(r"纳税人识别号|地址|电话|...")` 截断，但**没有列入"统一社会信用代码"**，导致名称后面粘着 `统一社会信用代码:91330000253917765N`。
- 增值税票的购买方名称是跨行表格排版（`购\n名称：xxx`），当前版 `_parse_name` 用 `购买方` 做 anchor，但实际文本用的是 `购` 与 `名称` 分两行的表格格式，anchor 匹配失败 → 返回空。

### 5.3 税号 `btax` / `vtax` — 命中率 0%（全部串为发票号码）

当前版 `_parse_tax_id` 的兜底逻辑是"全文找一处 15~20 位字母数字"，而发票号码恰好是 20 位数字，优先命中 → 所有税号都被解析为发票号码。

### 5.4 销售方 `vname` / `vtax` — 命中率 0%

- 铁路票本身不含销售方信息（承运方为中国铁路，不在电子客票文本中），属于 N/A 字段，不计入分母。
- 有销售方的发票（05/06/08）同样因税号兜底串位 + 名称跨行/锚点不匹配而全部失败。

### 5.5 分类 `cat` — 命中率 80%

- 滴滴行程单误判为"机票"：分类关键词中 `行程单` 命中 `flight` 类的"行程单"关键词。这是关键词设计问题——"行程单"同时适用于机票和网约车。

### 5.6 日期 `date` — 命中率 87.5%

- #06 日期 `2026年02月03日` 被解析为 `2000-00-13`：当前版 `_parse_date` 的正则 `(20\d{2})` 没有限制月/日的合理范围，导致匹配了 `2026` 年的年份 + `02` 月 + `03` 日中的某个数字 → 脱靶匹配到 `20`+`00`+`00` 或 `20`+`00`+`13`。这是正则贪婪匹配 + 缺乏上下文锚定导致的边界 Bug。

## 6. 改进验证：可达上限 91.8%

为验证"问题是否可修"，我用一个改进版解析器（仅用于评估，不替换交付代码）做了如下针对性修正：

| 修复点 | 修正方式 | 效果 |
|---|---|---|
| 金额 | 增加 `票价` 标签、`（小写）¥X` 模式、空格容错 `合\s*计`、滴滴 `合计X元` | 2/10 → 9/10 |
| 购买方名称 | 在截断规则中加入 `统一社会信用代码`；增加跨行 `购\s*名称` 匹配 | 0/8 → 7/8 |
| 税号 | 按顺序分配：第一个税号=购买方，第二个=销售方；取消"全文兜底匹配" | 0/7 → 7/7 |
| 分类 | 移除"行程单"→flight 映射（保留给航班专用关键词） | 8/10 → 9/10 |
| 日期 | 增加"开票日期"上下文锚定，避免脱靶 | 7/8 → 8/8 |

残余漏识别（3 个字段）：

- **#05 销售方税号**：发票仅含一个税号（购买方为个人"周扬"，无税号），启发式"第一个=买方"将卖方税号错配 → 需要上下文语义识别（如"销 统一社会信用代码"的位置信息）。
- **#08 销售方名称/金额**：客运票的版面极复杂，名称跨 3 行表格，价税合计与标签相距数行 → 需要版面感知（表格结构解析）或 LLM 辅助。
- **#09 分类**：目的地含"桔子酒店"触发 hotel 误判 → 需要更精细的分类策略（如发票类型优先于关键词）。

## 7. 其他发现

### 7.1 文本层：10 张 PDF 全部有文本层，无需 OCR

所有 PDF 通过 pdfplumber 均能抽出可读中文文本，原始文本长度在 248~696 字符之间。这意味着**当前 44.3% 的准确率不是 OCR 缺失导致的**，纯粹是正则规则设计问题。

### 7.2 OFD 文件完全无法处理

样本中 7 张 OFD 格式发票（与 PDF 一一对应或独立），当前 `extract_text()` 对 `.ofd` 扩展名直接返回空串（代码注释 `# ofd / 其他：暂不支持文本抽取`）。OFD 是国税总局推行的电子发票国家标准格式，大量企业已切换，需尽快支持。

### 7.3 XML 全量数据未被利用

#08 附带 `dzfp_26447100000001777638_20260212100055.xml`，XML 包含完整结构化发票数据（发票代码、号码、金额、税额、购销双方全字段），但当前解析器完全未使用。XML 解析可实现接近 100% 的准确率。

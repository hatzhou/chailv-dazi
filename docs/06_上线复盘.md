## 1. 需求目标达成评估框架

对照 PRD §2.2「产品目标（可量化）」与 §4「功能清单（6 大模块）」，从**指标达成、功能覆盖、体验约束**三个维度评估。评估口径需可计量、可追溯、可复核。

### 1.1 评估维度与口径

| 维度 | 评估对象 | 数据来源 | 达成判定口径 |
|---|---|---|---|
| D1 量化指标达成 | PRD §2.2 六项目标值 | 埋点/日志/导出记录/人工抽样 | 实测值 ≥ 目标值 ⇒ 达成；区间内浮动 ±2% 记为「基本达成」；低于阈值 ⇒ 未达成 |
| D2 功能覆盖率 | §4 的 6 大模块 + F 编号清单 | 需求-用例映射表、测试报告 | 覆盖率 = 已上线可用功能数 / P0+P1 功能总数 ×100%；P0 缺项直接判「未达成」 |
| D3 体验/约束符合 | §1.1 设计原则（本地优先、解析兜底、状态防重、金额口径） | 代码走查 + 用例 | 每条原则有对应实现与验证记录 ⇒ 符合；违反 ⇒ 严重问题入 §3 |
| D4 用户价值感知 | 用户故事 US-1~US-7 | 内测问卷/访谈 | NPS ≥ 30 或「核心场景可走通」占比 ≥ 80% ⇒ 价值成立 |

### 1.2 评估流程（四步）

1. **采集**：由研发导出解析日志、导出日志、导入流水；由测试导出用例通过率；由设计导出可用性走查。
2. **对照**：将实测值填入「PRD 目标达成对照表」（模板见下），标注 达成/基本达成/未达成。
3. **归因**：未达成项须填写根因（技术/需求/资源/外部依赖）与证据。
4. **结论**：输出「总体达成率」与「是否达到上线验收标准」结论，提交复盘会。

## 3. 遗留问题清单模板（按严重度）

按 **P0 致命 / P1 严重 / P2 一般 / P3 建议** 四级分级，P0/P1 需有「临时规避方案 + 修复排期」。

| 编号 | 标题 | 模块 | 严重度 | 现象描述 | 影响范围 | 根因（初步） | 临时规避 | 修复负责人 | 目标版本 | 状态 |
|---|---|---|---|---|---|---|---|---|---|---|
| BUG-001 | | | □P0 □P1 □P2 □P3 | | | | | | | □待处理 □修复中 □已修复 □已验证 |
| BUG-002 | | | | | | | | | | |

### 严重度定义
- **P0 致命**：导致数据丢失/无法使用/隐私泄露风险，需 24h 内响应、热修或临时下线相关能力。
- **P1 严重**：核心流程阻断（如某来源导入全失败、导出打不开），需 3 个工作日内修复。
- **P2 一般**：功能可用但体验受损或偶发错误，纳入下一迭代。
- **P3 建议**：优化类、文案/样式问题，可 backlog 排期。

### 填写纪律
- 每个问题必须可复现（附步骤/样本/日志），禁止「偶发但说不清」。
- 状态流转须闭环：待处理 → 修复中 → 已修复 → 已验证，禁止长期停留「修复中」。

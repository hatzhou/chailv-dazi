## 2. 研发排期建议（里程碑划分）

> 里程碑为**逻辑交付节点**，按依赖顺序推进；周期为相对排期建议（周），实际以周会燃尽调整。每个里程碑给出**目标 / 关键交付物 / 退出标准**。

### M1 · 数据层（基础，建议第 1~2 周）
- **目标**：打通「库 + 模型 + 仓储 + 工具」底座，让上层模块可并行开发。
- **交付物**：`db/database.py`、`db/migrations.py`（5 表 + 索引 + seed）、`db/repositories.py`、`utils/{path,date,crypto,font}_utils.py`、单测覆盖 CRUD。
- **退出标准**：应用启动自动建库入库成功；`InvoiceRepo.list` 在 1 万条桩数据下筛选 <200ms；相对路径存取单测通过。

### M2 · 采集 & 解析（建议第 2~4 周）
- **目标**：实现「导入→解析→去重→入库」主链路（本地 + 邮件 + 链接 + 手工；PDF 正则解析）。OCR/OFD 留接口但非必须。
- **交付物**：`importers/*`、`parsers/{base,pdf,classifier}`、占位 `ofd/image_parser`、`services/import_service.py`、`services/dedup.py`、解析回归集。
- **退出标准**：本地 PDF 批量导入自动抽字段；抽样 200 张准确率 ≥95%、单张 <1.5s（AC-2/AC-3）；去重跳过提示正确（AC-6）；扫描件标记「待补录」。

### M3 · 界面（建议第 3~6 周，与 M2 后段并行）
- **目标**：7 个主页面 + 三维组织 + 详情编辑 + 行程管理完整可用，四态齐备。
- **交付物**：`ui/main_window.py`、`ui/pages/*`（dashboard/invoice_list/invoice_detail/import_wizard/trip/settings）、`ui/widgets/*`、`ui/styles/*`(QSS)。
- **退出标准**：列表/搜索/筛选 <200ms（AC-7）；`reimbursed` 金额锁定（AC-5）；状态流转非法操作禁用；四态无白屏。

### M4 · 导出（建议第 5~7 周，与 M3 后段并行）
- **目标**：Excel/PDF 报销单 + 附件打包 + 导出后置状态。
- **交付物**：`exporters/{base,excel,pdf}_exporter`、`services` 导出编排、设置中心与邮件引导。
- **退出标准**：Excel 100% 可打开字段完整（AC-9）；PDF 可打开且明细一致（AC-10）；导出后「待报销」置「已提交」（AC-11）；附件缺失标注。

### M5 · 联调 & 验收（建议第 7~9 周）
- **目标**：端到端打通 + 全量 AC 验收 + 三平台验证 + 打包。
- **交付物**：跨模块联调、回归测试报告、`tests/` CI 绿、Windows/macOS/Linux 字体与路径验证、PyInstaller 出包（--onedir，解 R-T3）。
- **退出标准**：13 条 AC 全部通过（AC-1~AC-14，含 AC-12 不上云、AC-14 金额口径）；三平台启动正常无 CJK 方块；导出包体积与启动时间在可接受范围。

## 3. 技术方案评审结论

> 结论基于《03_设计研发评审.md》§2/§4 的议题与决策，作为编码依据。

### 3.1 SQLite 选型结论
- **结论：确认采用 SQLite 单文件库**，符合「本地优先、数据不上云」原则（解 PN-6 / R-C3）。
- **注意事项**：
  1. **并发模型**：单写者队列串行化写操作 + `PRAGMA journal_mode=WAL`，读多写少场景开启 WAL 可显著降低 `database is locked`（解 R-D3）。
  2. **金额精度**：阶段一用 `REAL`（元），若报销精度争议升级为「分」整数存储需迁移；本期已在 models 用 `amount/tax_amount/amount_excl_tax`。
  3. **去重策略**：对「非空代码 + 号码」建**部分唯一索引**（§2.2 `uq_invoices_code_no`），手工票放行避免误合并（解 R-D1）。
  4. **附件路径**：`attachments.rel_path` 存**相对 DATA_DIR 的相对路径**，访问时 `path_utils` 拼当前 DATA_DIR，迁移整体拷贝（解 R-D2）。
  5. **冗余字段**：`category_name/category_icon/trip_name` 为查询性能冗余，分类/行程改名时需在 Repo 级联刷新。
  6. **安全**：敏感键（imap_pass）只存密文（Q-4）；库损坏启动校验并提示用备份恢复。

### 3.2 解析策略结论（正则为主 + OCR 可选）
- **结论：阶段一以「pdfplumber 抽文本 + 正则抽取」为主路径；OCR 仅留 `ImageParser` 接口位，不纳入阶段一（Q-1 决议）。OFD 仅存储不解析（Q-5）。**
- **注意事项**：
  1. **置信度阈值**：默认 `confidence < 0.8` 标记 `need_review=1` 标黄进入人工复核（T-203/§2.4.2 `confidence_of`）；阈值可经 T-4 评审微调并写进设置。
  2. **解析兜底**：`Parser.parse` 约定「不抛异常、缺失字段返回 warnings」，确保导入向导逐条标红不中断（I-2）；原文件 100% 保留（AC-4）。
  3. **扫描件处理**：检测文本长度过短 → `confidence=0` + warning「疑似图片 PDF，请走手工/后续 OCR」。
  4. **回归集**：`tests/fixtures/` 固定 20+ 真实版式发票（火车/机票/酒店/打车/餐饮/增值税专票），CI 断言准确率 ≥95%，防止正则回归（解 R-T1）。
  5. **分类可配置**：预置 `DEFAULT_CATEGORIES` + 允许用户增/改/排序（Q-8 决议），`classifier` 关键词命中需与设置中分类保持一致。

### 3.3 PySide6 界面架构结论
- **结论：分层 + 信号槽 + 后台线程**，依赖方向 `ui → services → {importers,parsers,exporters} → db → models/config`，**禁止反向依赖**（db 不引 ui，importers 不直接操作 UI，仅进度回调）。
- **注意事项**：
  1. **线程模型**：所有下载/解析/导出进 `QRunnable`/`QThread` 线程池，主线程只更新模型，禁止主线程同步 IO（解 R-T2）；进度用信号/`QMetaObject.invokeMethod` 跨线程刷新（I-3）。
  2. **状态机驱动**：发票状态机 `draft→pending→submitted→reimbursed→rejected` 的合法迁移在 UI 层**禁用非法操作**（§10.3）；`reimbursed` 锁定金额编辑（AC-5）。
  3. **图表**：用 QtCharts 绘制仪表盘，CJK 字体经 `font_utils` 运行时探测注册（Win msyh / macOS PingFang / Linux Noto/WenQuanYi），避免方块（解 §4.4）。
  4. **样式与四态**：QSS 主题统一；每页实现空/加载/错误/空数据四态，Error 给可操作下一步、Empty/NoResult 给主行动按钮（设计验收点 §3.4）。
  5. **打包**：PyInstaller `--onedir` 优先，`hooks` 处理 pdfplumber/pandas 隐式依赖，三平台分别出包（解 R-T3）。

## 4. 风险识别与应对

> 风险分三类：**范围变更 / 延期风险 / 阻塞事项**。每项给「风险 → 影响 → 应对」。

### 4.1 范围变更风险
| 风险 | 影响 | 应对 |
|---|---|---|
| Q-1 OCR 是否纳入（图片发票自动解析） | 若中途加入，引入 paddleocr/tesseract 重依赖（>数百 MB），冲击打包与排期 | 阶段一按 Q-1 决议：**仅手工录入**，OCR 仅留接口位；如确需，单列里程碑并冻结合约 |
| Q-4 授权码存储方式 | 明文落盘=合规红线 | 落地 `crypto_utils`（keyring 优先，回退本地 AES 加密）；日志禁打授权码/税号 |
| Q-5 OFD 解析深度 | 引入 ofdparser 与版式解析 | 阶段一仅存储不解析，`OfdParser` 占位；如需解析单独立项 |
| Q-8 分类可配置性 | 影响 categories 表与设置 UI 工作量 | 预置 + 用户增改排序均已规划（T-203/T-602），按评审落地即可 |
| Q-3 导出模板对齐具体财务规范 | 字段/版式返工 | 先通用模板 + `ExportOptions.template` 预留自定义，对接具体公司时增量 |

**范围护栏**：任何 Q-x 决议变更须经研发负责人 + 产品双签，更新本表与 PRD 修订记录，禁止研发私自扩大范围。

### 4.2 延期风险
| 风险 | 影响 | 应对 |
|---|---|---|
| 解析准确率不达标（<95%） | 标黄过多，用户仍手工抄录，价值落空 | 解析回归集 + CI 门禁；按票种逐组调正则；责任人 T-202/T-205 |
| 批量导入（>500 张）卡 UI | 体验崩塌、被误判为死机 | 严格后台线程 + 分批队列 + 总进度条（T-301，解 R-T2） |
| 打包体积/启动慢（>200MB） | 分发与首启体验差 | --onedir + 裁剪隐式依赖；非阻塞项，M5 处理（R-T3） |
| 三平台字体/路径差异 | 跨平台出方块或路径失效 | font_utils + path_utils 提前在 M1 落地并单测（解 §4.4） |

### 4.3 阻塞事项
| 阻塞项 | 阻塞范围 | 应对 / 解锁条件 |
|---|---|---|
| **OCR 依赖未定**（Q-1） | 图片发票自动解析 | 阶段一不阻塞：走手工录入 + 标黄；OCR 作为 Could 后置，接口已预留 |
| **邮件协议兼容性**（IMAP SSL/端口/授权码） | 邮箱导入（F-1.2） | `COMMON_IMAP_HOSTS` + `imap_ssl` 开关 + 连接测试给具体错误；QQ 需 SSL 993 等差异已枚举；不删服务器邮件 |
| **跨平台路径/字体** | 全模块存取与图表渲染 | `path_utils`（全程 pathlib，DB 只存相对路径）+ `font_utils`（运行时探测 CJK）；M1 即落地 |
| **数据库写锁**（并发） | 高并发导入 | 单写者队列 + WAL（T-000-1，解 R-D3），M1 即解决 |
| **附件路径失效**（改目录/迁移） | 详情预览打不开 | 相对路径 + 迁移工具整体拷贝 DATA_DIR（T-601，解 R-D2） |

> 升级机制：阻塞项超过 2 个工作日未解锁，由研发负责人拉齐产品/安全拍板；范围类变更写入 §4.1 护栏流程。

## 5. 进度跟踪机制

### 5.1 看板字段定义
- **看板列（状态流转）**：`Backlog → To Do → In Progress → Code Review → Testing → Done`。
- **卡片必填字段**：

| 字段 | 取值/说明 |
|---|---|
| 任务编号 | 如 T-202（对应 §1 拆解） |
| 标题 | 一句话任务，如「PDF 正则抽取引擎」 |
| 模块 | 1 多源采集 / 2 解析 / 3 管理 / 4 仪表盘 / 5 导出 / 6 本地化 / 0 基础 |
| 负责人 | 实际开发同学（Arch/BE/FE/QA/SEC） |
| 优先级 | P0 Must / P1 Should / P2 Could（沿用 PRD MoSCoW） |
| 预估 / 剩余工作量 | 人日，每周更新剩余 |
| 依赖 | 前置任务编号；未解依赖禁止进 In Progress |
| 关联文档 | PRD 章节（如 F-2.1）/ 设计稿 / 评审结论（如 §2.4.2） |
| 验收标准 | 对应 AC 编号（如 AC-2）或自测点 |
| 风险标记 | 🔴阻塞 / 🟡有风险 / 🟢正常（每周滚动） |
| 进展备注 | 关键决策、偏离、需协调事项 |

- **WIP 限制**：`In Progress` 每人 ≤ 2 张，避免上下文切换；`Code Review` 停留 ≤ 2 天。

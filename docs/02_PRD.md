## 1. 背景（Background）

频繁出差职场人（销售/商务、项目驻场/实施顾问、行政财务对接人）面临发票来源分散（邮件 PDF、微信文件、纸质拍照）、手工抄录字段低效易错、月底集中归集耗时且易漏报/重报、缺少按"行程+分类"的成本归集、导出报销表格式不统一被财务反复打回等痛点。

《01_需求规划.md》已确认：
- **阶段一定位**：个人单机使用，本地优先、数据不上云（解决隐私痛点 PN-6）。
- **核心能力范围**：多源采集、智能解析、集中管理、仪表盘、报销导出、数据本地化 6 大模块。
- **MoSCoW 取舍**：Must（必须有）/ Should（尽量做）/ Could（富余做）/ Won't（阶段一不做）。

本文档将上述规划转化为可落地需求。下表为术语与上游映射，便于追溯。

### 1.1 设计原则（约束基线）
1. **本地优先**：所有发票数据存于本机 SQLite 单文件，附件落盘于本地目录，严禁静默上传云端。
2. **解析兜底**：自动解析失败时不得丢文件、不得丢数据，必须可人工补录。
3. **状态防重**：依靠状态机 + 去重，将漏报/重报率趋近于 0。
4. **金额口径统一**：系统内一切金额以"价税合计"为准（详见 §11.3）。
5. **范围克制**：阶段一不做团队协同、不做 OA/财务系统对接、不做云端同步（Won't）。

## 2. 目标（Goals）

### 2.1 业务目标
- 将用户"月底报销准备"从数小时压缩到分钟级，减少漏报/重报。
- 以"本地隐私 + 自动解析 + 行程归集 + 规范导出"建立差旅场景下的发票管理第一站。

### 2.2 产品目标（可量化，阶段一验收参考）
| 目标 | 目标值 | 对应上游 |
|---|---|---|
| PDF 关键字段自动解析准确率 | ≥ 95% | §4 定量目标 |
| 单张 PDF 解析耗时（抽取+正则） | < 1.5s | §4 定量目标 |
| 月底报销准备耗时 | 较手工 ↓ 70% | PN-3 |
| 漏报/重报发生率 | 趋近于 0 | PN-3 |
| 支持发票来源 | ≥ 3 种（本地/邮件/手工；链接 Should） | R-01/02/03/04 |
| 导出成功率 | 100%（可打开、字段完整） | R-14/15 |

## 4. 功能清单（Feature List，6 大模块）

> 优先级沿用 MoSCoW：P0=Must / P1=Should / P2=Could / —=Won't(阶段一)。每条关键需求在 §4.x 内以 **[EARS类型]** 标注。  
> EARS 类型：Ubiquitous（始终）/ Event-driven（当…时）/ Unwanted（若…则）/ State-driven（当状态为…时）/ Optional（若用户配置…则）。

### 模块一：多源采集（R-01~R-04）
| 编号 | 功能 | 说明 | 优先级 | 上游 |
|---|---|---|---|---|
| F-1.1 | 本地文件批量导入 | 文件多选对话框，支持 PDF/OFD/图片 | P0 | R-01 |
| F-1.2 | 邮件 IMAP 导入 | 配置主机+授权码，拉取带附件邮件并解析 | P1 | R-02 |
| F-1.3 | 链接 URL 下载导入 | 粘贴 URL，下载到附件目录并解析 | P1 | R-03 |
| F-1.4 | 手工录入 | 全字段表单录入，兜底无法解析场景 | P0 | R-04 |

**关键需求（EARS）**
- **[Ubiquitous]** 系统应始终在导入发票时校验文件类型是否为 PDF/OFD/图片，非支持类型应拒绝并提示。
- **[Event-driven]** 当用户从邮件导入发票时，系统应拉取近 N 天（默认 30，可配置）含附件的邮件并解析。
- **[Event-driven]** 当用户粘贴发票链接并确认下载时，系统应使用 requests 下载文件至附件目录并进入解析流程。
- **[Event-driven]** 当用户提交手工录入表单时，系统应校验必填字段（发票代码、号码、价税合计、开票日期）完整后方可入库。
- **[Unwanted]** 若下载链接失效或返回非发票文件，则系统应提示失败原因并保留该来源记录为"待补录"。
- **[Optional]** 若用户配置了 OCR 引擎，则系统应对图片类发票执行文字识别后再走解析流程。

### 模块二：智能解析（R-05~R-08）
| 编号 | 功能 | 说明 | 优先级 | 上游 |
|---|---|---|---|---|
| F-2.1 | PDF 文本解析 | pdfplumber 抽文本 + 正则抽取字段 | P0 | R-05 |
| F-2.2 | 自动分类 | 按关键词/版式归类（火车票/机票/酒店/打车/餐饮等） | P0 | R-06 |
| F-2.3 | 解析结果可编辑 | 列表/详情字段可改，失败标黄 | P0 | R-07 |
| F-2.4 | 图片 OCR 解析 | 集成 OCR 引擎识别后解析（可选） | P2 | R-08 |

**关键需求（EARS）**
- **[Ubiquitous]** 系统应始终在解析完成后回填发票代码、号码、价税合计、税额、不含税金额、开票日期、购销方名称与税号字段。
- **[Ubiquitous]** 系统应始终对解析所得字段做基本合理性校验（如价税合计 ≥ 不含税金额、日期格式合法）。
- **[Event-driven]** 当 PDF 文本抽取完成时，系统应应用正则规则抽取关键字段并触发自动分类。
- **[Unwanted]** 若解析失败或字段缺失，则系统应保留原文件并标记该发票为"待补录"（标黄），不丢弃数据。
- **[Optional]** 若用户配置了 OCR 引擎，则系统应对图片类发票执行文字识别并复用同一套字段抽取规则。
- **[Event-driven]** 当自动分类置信度低于阈值时，系统应将该发票归入"未分类"并提示用户手动指定。

### 模块三：集中管理（R-09~R-12）
| 编号 | 功能 | 说明 | 优先级 | 上游 |
|---|---|---|---|---|
| F-3.1 | 三维组织（行程/分类/状态） | 树/标签式组织 | P0 | R-09 |
| F-3.2 | 搜索/筛选/排序 | 关键字+多条件组合 | P0 | R-09 |
| F-3.3 | 详情编辑 | 单条字段编辑、附件查看 | P0 | R-09 |
| F-3.4 | 发票状态机 | draft→pending→submitted→reimbursed/rejected | P0 | R-10 |
| F-3.5 | 行程管理 | 新建/编辑行程，挂接发票 | P0 | R-11 |
| F-3.6 | 发票去重 | 按 代码+号码 去重 | P1 | R-12 |

**关键需求（EARS）**
- **[Ubiquitous]** 系统应始终允许按发票代码+号码检索，且金额汇总以"价税合计"累加。
- **[State-driven]** 当发票状态为"已报销"时，系统应锁定金额字段不可编辑（仅允许备注/状态回退）。
- **[State-driven]** 当发票状态为"已提交"时，系统应在导出勾选中默认排除，防止重复提交。
- **[Event-driven]** 当用户将发票挂接到某行程时，系统应实时更新该行程的归集金额与预算使用率。
- **[Unwanted]** 若用户尝试导入与已有发票代码+号码完全相同的记录，则系统应提示重复并跳过建记录（保留原记录）。
- **[Event-driven]** 当用户执行搜索/筛选时，系统应在 200ms 内返回结果（本地 SQLite 索引查询）。

### 模块四：仪表盘（R-13）
| 编号 | 功能 | 说明 | 优先级 | 上游 |
|---|---|---|---|---|
| F-4.1 | 总额 / 待报销额 | 关键数字卡片 | P1 | R-13 |
| F-4.2 | 分类占比 | 饼/环图 | P1 | R-13 |
| F-4.3 | 行程汇总 | 按行程金额 vs 预算 | P1 | R-13 |
| F-4.4 | 月度趋势 | 折线/柱状 | P1 | R-13 |

**关键需求（EARS）**
- **[Ubiquitous]** 系统应始终在发票新增/编辑/状态变更后刷新仪表盘聚合数据。
- **[Event-driven]** 当用户切换时间范围筛选时，系统应重算总额、待报销额与月度趋势。
- **[State-driven]** 当某行程归集金额超过预算时，系统应在行程汇总中以预警色标识。

### 模块五：报销导出（R-14~R-15）
| 编号 | 功能 | 说明 | 优先级 | 上游 |
|---|---|---|---|---|
| F-5.1 | Excel 明细导出 | pandas+openpyxl 生成规范明细表 | P0 | R-14 |
| F-5.2 | PDF 报销单导出 | 封面+明细，可附原始附件打包 | P1 | R-15 |

**关键需求（EARS）**
- **[Event-driven]** 当用户勾选发票并触发 Excel 导出时，系统应生成含全部关键字段的明细表（代码/号码/价税合计/税额/销售方等）。
- **[Event-driven]** 当用户触发 PDF 报销单导出时，系统应生成封面+明细，并可选打包原始附件。
- **[State-driven]** 当导出动作完成时，系统应将所选"待报销"发票状态置为"已提交"，避免重复导出。
- **[Unwanted]** 若导出过程中附件缺失或损坏，则系统应继续生成报表并在备注列标注"附件缺失"。

### 模块六：数据本地化与设置（R-16~R-18）
| 编号 | 功能 | 说明 | 优先级 | 上游 |
|---|---|---|---|---|
| F-6.1 | 本地持久化 | SQLite 单文件 + 附件落盘 | P0 | R-16 |
| F-6.2 | 设置中心 | 数据目录/IMAP/导出偏好 | P1 | R-17 |
| F-6.3 | 引导式邮件配置 | 主机下拉+连接测试 | P1 | R-18 |

**关键需求（EARS）**
- **[Ubiquitous]** 系统应始终将发票结构化数据与附件仅存储于本机用户指定目录，不上传任何远程服务器。
- **[Event-driven]** 当用户在设置中修改数据目录时，系统应校验目录可写并迁移已有库与附件。
- **[Event-driven]** 当用户完成邮件配置并点击"连接测试"时，系统应验证 IMAP 主机/授权码可用性并反馈结果。
- **[Optional]** 若用户启用自动备份，则系统应定期复制 SQLite 文件至备份目录。

## 5. 流程说明（Process）

### 5.1 多源采集主流程
```
[开始]
  │
  ├─ 本地导入 ──► 文件选择对话框(多选 PDF/OFD/图片)
  ├─ 邮件导入 ──► 校验IMAP配置 ──否──► 引导配置 ──► 连接测试
  │                │是
  │                ▼
  │              拉取近N天带附件邮件 ──► 下载附件到附件目录
  ├─ 链接导入 ──► 粘贴URL ──► requests下载 ──失败──► 标记"待补录"
  └─ 手工录入 ──► 表单填写全字段
        │
        ▼
  文件类型校验(PDF/OFD/图片?) ──否──► 拒绝并提示
        │是
        ▼
  写入"待解析"队列 ──► 进入[解析流程]
```

### 5.2 解析流程
```
[解析开始]
  │
  ├─ PDF/OFD ──► pdfplumber抽取文本
  │                │
  │                ▼
  │              正则抽取字段(代码/号码/价税合计/税额/不含税/日期/购销方/税号)
  ├─ 图片 ──► OCR已配置? ──是──► 文字识别 ──► 同上正则
  │             │否
  │             ▼
  │           标记"待补录"(需手工)
  │
  ▼
字段合理性校验(价税合计≥不含税? 日期合法?)
  │
  ├─ 校验失败 / 字段缺失 ──► 标记"待补录"(标黄) + 保留原文件
  ├─ 置信度低(分类) ──────► 归入"未分类" + 提示手动指定
  └─ 成功 ──────────────► 自动分类 + 去重检查
                              │
                              ├─ 重复(代码+号码) ──► 跳过,提示
                              └─ 新记录 ──► 状态="待报销" 入库
```

### 5.3 导出流程（简述）
```
勾选发票(或按行程/状态筛选全选) ──► 生成Excel明细 + 可选PDF报销单
  │                                    │
  ▼                                    ▼
所选"待报销"置"已提交" ──────────► 附件打包(可选) ──► 完成
```

## 6. 交互说明（Interaction，关键页面）

### 6.1 导入页
- 顶部 Tab 切换：本地 / 邮件 / 链接 / 手工。
- 本地：拖拽或按钮多选；下方实时进度条与"成功 N / 待补录 M"计数。
- 邮件：主机下拉（QQ/163/Gmail…）+ 授权码输入框 + "连接测试"按钮；拉取前显示预估邮件数。
- 手工：结构化表单，必填项红星；价税合计失焦时自动反算税额/不含税（若可）。

### 6.2 集中管理页（列表）
- 三维组织：左侧树（行程 ▸ 分类 ▸ 状态）；右侧表格。
- 表格列：缩略图、购销方、价税合计、税额、开票日期、分类、状态、来源。
- 行交互：单击看详情；双击进编辑；标黄行=待补录。
- 批量栏：全选 / 按状态筛选 / 搜索框（关键字+日期区间+金额区间）/ 列排序（点击表头）。
- 状态切换：下拉或右键菜单流转（受 §10 状态规则约束）。

### 6.3 详情/编辑抽屉
- 左：原始附件预览（PDF 翻页 / 图片缩放）。
- 右：字段表单，可编辑；"已报销"状态锁定金额相关字段（置灰）。
- 底部：挂接行程下拉、状态切换、保存/取消。

### 6.4 仪表盘页
- 顶部 4 数字卡：总金额、待报销额、本月新增、已报销额。
- 中部：分类占比（环图）+ 行程汇总（条形，超预算预警色）。
- 下部：月度趋势（折线，可切年/月）。
- 时间范围筛选器联动全部图表。

### 6.5 导出页
- 来源：从集中管理勾选带入，或"按筛选结果全选"。
- 选项：导出 Excel / 导出 PDF / 打包附件（checkbox）。
- 预览：导出字段清单 + 记录数 + 合计金额；点"导出"选路径。

## 7. 数据指标（Metrics）

### 7.1 北极星指标（North Star）
**「月活用户人均通过系统自动归集并成功导出的发票数」**  
理由：该指标同时衡量"自动解析闭环的使用深度"与"报销价值兑现"，与产品核心目标（归集+导出）直接挂钩，且可本地埋点统计（见 §12）。

### 7.2 过程指标（Process Metrics）
| 指标 | 定义 | 目标 | 用途 |
|---|---|---|---|
| 解析准确率 | 自动填充关键字段正确数 / 抽样总数 | ≥ 95% | 监控解析质量 |
| 单张解析耗时 | PDF 抽取+正则平均耗时 | < 1.5s | 性能基线 |
| 自动归集覆盖率 | 自动解析成功数 / 导入总数 | 越高越好 | 衡量 OCR/分类价值 |
| 待补录占比 | 待补录数 / 总数 | 越低越好 | 发现解析短板 |
| 导出成功率 | 成功打开且字段完整 / 导出次数 | 100% | 导出质量 |
| 重复拦截数 | 去重跳过的导入次数 | 监控重报防护 | 状态机有效性 |
| 报销准备耗时压缩比 | (手工基线 − 系统耗时) / 手工基线 | ≥ 70% | 价值验证 |

## 9. 边界场景（Boundary Scenarios）

## 10. 异常状态（Exception States）

### 10.1 解析异常
- 文本抽取失败 → 标记"待补录"，原文件留存，错误日志记原因。
- 正则未命中关键字段 → 仅填充命中项，缺失项标黄，状态"待补录"。
- OCR 引擎未安装却被调用 → 禁用图片解析入口并提示"未配置 OCR"。

### 10.2 网络异常（仅邮件/链接）
- IMAP 连接超时/鉴权失败 → 连接测试返回明确错误，不写入脏数据。
- 链接 4xx/5xx 或超时 → 标记来源记录"待补录"，提示失败原因。

### 10.3 状态机异常（防重与回退）
状态集合：`draft 草稿 / pending 待报销 / submitted 已提交 / reimbursed 已报销 / rejected 被退回`。
- 合法正向：draft→pending→submitted→reimbursed；reimbursed→rejected（财务退回）→pending。
- **非法回退拦截**：reimbursed 不得直接回退到 draft/pending（需经 rejected）；submitted 不得改金额。
- 任何状态变更须记录操作时间，供审计。

### 10.4 存储异常
- SQLite 损坏 → 启动时校验，损坏则提示用备份恢复（§6.3 自动备份）。
- 附件文件丢失（被外部删除）→ 详情页预览提示"附件缺失"，不影响结构化数据。

## 11. 权限规则与数据口径（Rules & Definitions）

### 11.1 权限规则
- 阶段一为**单机个人版**，无多账号；应用级无角色权限差异。
- 数据访问范围：仅当前系统用户本机目录，不涉及操作系统级多用户隔离（如需可后续增强）。
- 敏感信息（税号、授权码）：IMAP 授权码优先存系统钥匙串（Q-4），不可明文落日志。
- 导出文件含敏感税号，导出目录须为用户显式选择，不做自动外发。

### 11.2 字段口径（核心数据模型要点）
| 字段 | 说明 | 必填 | 备注 |
|---|---|---|---|
| invoice_code 发票代码 | 纸质/电子发票代码 | 是* | 电子普票可能为空，允许空但提示 |
| invoice_no 发票号码 | 唯一性组成 | 是 | 与代码构成去重键 |
| total_amount 价税合计 | **金额口径基准** | 是 | 所有汇总以此为准 |
| tax_amount 税额 | 税额 | 否 | total = taxable + tax |
| taxable_amount 不含税金额 | 不含税 | 否 | |
| invoice_date 开票日期 | ISO 日期 | 是 | |
| seller_name / seller_tax_no | 销售方名称/税号 | 否 | 解析尽量抽取 |
| buyer_name / buyer_tax_no | 购买方 | 否 | 多购买方场景 |
| category 分类 | train/air/hotel/taxi/meal/other | 是 | 默认 other |
| status 状态 | 见 §10.3 | 是 | 默认 pending |
| source 来源 | local/email/url/manual | 是 | |
| trip_id 行程 | 外键可空 | 否 | |

### 11.3 金额口径（统一规则）
- **系统内所有合计、占比、趋势、导出金额均以 `total_amount`（价税合计）为准**。
- 校验关系：`total_amount ≥ taxable_amount`（允许相等，即零税率）；税额缺省时按 `tax = total − taxable` 反算填充。
- 编辑时若用户改价税合计，提示是否同步重算税额/不含税（或留空待补）。

### 11.4 币种处理
- 默认记账币种 **CNY**；发票记录支持 `currency` 字段（ISO 4217，如 USD）。
- 原币种金额 `total_amount` 按原币种存储；汇总时：
  - 同币种直接累加；
  - 跨币种汇总以记账币种（CNY）为基准，按**导出/查看时的固定汇率表**折算（阶段一用内置静态汇率，Q-待确认是否接实时汇率）；
  - 仪表盘与导出须标明币种，避免混合误导。
